graphify, commit by commit

A working-notes piece on treating graphify output as a first-class git artifact: what the tool does, where it fits in the commit / checkout loop, and how to fake AS OF <branch> time-travel over your code graph without a database.

What graphify actually is

graphify (the safishamsi/graphify project, package graphifyy on PyPI) takes a folder and produces a knowledge graph of what is in it. For a Java / Python / TS repo the interesting part is the code pass: tree-sitter parses files into ASTs, a deterministic walker pulls classes, functions, imports, and call edges, and a second pass adds inferred cross-file calls edges. Docs, PDFs, and images go through an LLM subagent pass; video and audio are transcribed locally with faster-whisper. Everything merges into a NetworkX graph, Leiden clusters it by edge density, and the output lands in graphify-out/.

Three files matter in practice:

  • graphify-out/graph.json: the canonical structure. Nodes with id, label, source_file, source_location; edges with relation (calls, imports, uses, etc.) and confidence tagged EXTRACTED, INFERRED, or AMBIGUOUS.
  • graphify-out/GRAPH_REPORT.md: one page, god nodes first, then communities and surprising connections. This is what AGENTS.md / CLAUDE.md point assistants at.
  • graphify-out/graph.html: a vis.js viewer, pan / zoom / search, opens in any browser.

Confidence tags are the bit that keeps it honest. EXTRACTED means it is literally in the source (an import, a direct call site). INFERRED is a call-graph second pass or a co-occurrence guess, with a numeric confidence_score on the edge. AMBIGUOUS surfaces in the report for human review. You can filter on this when you query.

Two design choices under the hood matter for how this lands in practice. Docstrings and rationale comments (# WHY:, # NOTE:, # HACK:, # IMPORTANT:) are pulled out as rationale_for nodes, so the graph captures not just what the code does but why it was written that way. And clustering is Leiden over the graph topology itself, with no embeddings and no vector store: the semantically_similar_to edges the LLM extracts are already in the graph, so they shape community detection directly. That is what makes per-commit rebuilds cheap on the structural side: tree-sitter AST (25 languages) plus NetworkX + Leiden recluster is milliseconds, and the LLM expense sits entirely in the doc / PDF / image pass.

The concrete setup (ground truth)

Install via pipx so the CLI is isolated from project venvs:

pipx install graphifyy
# binary lands at $HOME/.local/bin/graphify

Inside a repo (here: platform-capabilities, scoped to reporting-only work on a poc branch), three files do most of the configuration work:

# .graphifyignore  (same syntax as .gitignore)
# Narrow to the reporting surface so the graph is not dominated by unrelated modules.
node_modules/
target/
build/
dist/
**/generated/**
# everything except reporting:
/platform-treasury/
/platform-payments/
/platform-connectivity/
# keep:
!/platform-reporting/
# AGENTS.md  (committed; read by OpenCode and other AGENTS-aware assistants)
## graphify

A knowledge graph of this repo lives under `graphify-out/`.
Before searching raw files for architecture questions, read
`graphify-out/GRAPH_REPORT.md`. For specific traversals use:

    graphify query "..." --graph graphify-out/graph.json
    graphify path  "NodeA" "NodeB"
    graphify explain "NodeName"

Prefer graph edges over keyword grep when answering "what calls X"
or "where is Y defined".

Then the OpenCode plugin, installed with graphify opencode install, drops a tool.execute.before hook in .opencode/plugins/graphify.js and registers it in opencode.json. Before any bash tool call the plugin checks whether graphify-out/graph.json exists and injects a reminder into the tool output, so the assistant is nudged toward the graph rather than grep -r.

Git hooks come from graphify hook install, which writes post-commit and post-checkout hooks. Guard them to the branch you care about so a drive-by checkout of main does not trigger a rebuild:

# .git/hooks/post-commit  (snippet, after `graphify hook install`)
#!/usr/bin/env bash
set -e
branch=$(git rev-parse --abbrev-ref HEAD)
[ "$branch" = "poc" ] || exit 0
"$HOME/.local/bin/graphify" update . --no-viz >/dev/null 2>&1 || {
  echo "graphify: rebuild failed on $(git rev-parse --short HEAD)" >&2
  exit 1
}

Same pattern for post-checkout, with the branch check comparing the new HEAD. Non-zero exit on failure is deliberate: a silent stale graph is worse than an obvious broken one.

In a real repo the shapes are bigger and noisier, but the mechanic is the same. For scale, the project'\''s own worked examples give concrete shapes: a 6-file HTTP client comes in at 144 nodes / 330 edges / 6 communities; a 52-file mixed corpus (Karpathy repos + 5 papers + 4 diagrams) lands at 285 nodes / 340 edges / 53 communities with a ~71x query-token reduction (~1.7k tokens per query vs ~123k reading raw files). Those numbers are self-reported; treat them as directional, not a spec. Committing graphify-out/ (minus the cache/ subdir) means every commit hash maps one-to-one to a graph snapshot, and git log -- graphify-out/GRAPH_REPORT.md gives you a narrative of structural change that is separate from the line-diff narrative.

Per-commit vs per-branch: two different questions

Both modes have a place; they answer different questions.

per commit

Rebuild after every commit on the working branch. Fast because the AST pass is cheap and the SHA256 cache short-circuits unchanged files. Good for: bisecting structural regressions, watching a refactor land over many small commits, keeping the always-on assistant context in sync with HEAD.

per branch

Rebuild on branch switch, then snapshot the result into a per-branch directory (see below). Good for: comparing two long-lived lines of work, answering "how did the graph look at the tip of poc last Friday", or driving a poor-man'\''s AS OF query.

Two subtleties worth calling out:

  • The rebuild writes back into the working tree at graphify-out/. If you then git add graphify-out/ and amend, you have a fixed point: commit hash references its own graph. Cleaner, but costs an amend on every commit. Most setups skip this and simply commit the graph on a cadence, or only on merges into the integration branch.
  • The post-checkout hook fires on file checkouts too, not only branch switches. The first positional argument is 0 for a file checkout and 1 for a branch checkout; guard on it or you will rebuild on every git checkout -- some-file.
# .git/hooks/post-checkout
#!/usr/bin/env bash
prev_head=$1
new_head=$2
is_branch=$3
[ "$is_branch" = "1" ] || exit 0       # ignore file checkouts
branch=$(git rev-parse --abbrev-ref HEAD)
[ "$branch" = "poc" ] || exit 0        # scope to the working branch
"$HOME/.local/bin/graphify" update . --no-viz || exit 1

# snapshot per branch for AS OF queries
snap=".graphify-snapshots/$branch"
mkdir -p "$snap"
cp graphify-out/graph.json       "$snap/graph.json"
cp graphify-out/GRAPH_REPORT.md  "$snap/GRAPH_REPORT.md"

Poor man'\''s AS OF: branch snapshots as a time axis

graphify does not know about time. It knows about the filesystem it was pointed at. But git already has a time axis, and branches are named handles on that axis. If you copy graphify-out/graph.json into .graphify-snapshots/<branch>/ after every rebuild, you can answer questions of the form "what did the code graph look like as of the tip of <branch>" by loading the right file.

The query side is then a small wrapper:

graphify query "what calls ReportBuilder?" \\
  --graph .graphify-snapshots/poc/graph.json

# diff two snapshots
diff <(jq -S '.nodes|map(.id)|sort' .graphify-snapshots/main/graph.json) \\
     <(jq -S '.nodes|map(.id)|sort' .graphify-snapshots/poc/graph.json)

This is not a database and it is not transactional. It is four lines of shell and a directory. But you get the one property that matters: a stable, reproducible graph per branch that the assistant can read, the CLI can query, and you can diff.

The interesting read is not the count of added nodes (that roughly tracks lines of code). It is the added edges that cross community boundaries. A poc branch that introduces a new reporting adapter and connects it to an existing payments cluster via an INFERRED call is a structural signal that merge review should linger on that seam.

Two edge shapes are especially worth diffing on top of the usual calls / imports set. semantically_similar_to edges that appear only on poc flag code solving the same problem without calling each other, which is often a duplication smell introduced by the branch. And hyperedges, which group 3+ nodes into a single relation (all classes implementing a shared protocol, all concepts from one paper section), are a compact way to see whether a refactor pulled new members into an existing group or invented a new one. jq over graph.json handles both.

Practical workflows

Bisecting a structural regression

Somewhere in the last twenty commits, a core abstraction grew a dependency it should not have. Standard git bisect over tests will not catch it. Instead, at each bisect step, run graphify path "Bad" "ShouldNotReach" on the snapshot and let a non-zero hop count be the "good" signal. A minute per step, converges in log2(N).

Review aid for long-lived branches

On PR open, a CI job checks out both base and head, runs graphify update on each, and posts the node / edge diff as a PR comment. You see "3 new god nodes, 11 new cross-community edges" before you read any line diff. Cheap because the AST pass is deterministic and does not need the LLM.

Agent context scoping

.graphifyignore earns its keep here. On a monorepo, a full graph is noisy and expensive. Scoping to one functional area (the reporting-only pattern above) keeps GRAPH_REPORT.md focused so the assistant'\''s always-on context actually helps. The patterns work even when the tool is run from a subfolder, so one root file handles multiple entry points.

Asking "what connects A to B"

The path and explain commands are the high-value ones in day-to-day work:

graphify path "ReportBuilder" "JdbcReportWriter"
graphify explain "ReportBuilder"

Output includes relation types, confidence tags, and source locations, so it is a good intermediate block to hand to an LLM without pasting the whole repo.

Pitfalls

  • Inferred edges lie sometimes. The INFERRED tag is there because the call-graph second pass can wire up false positives, especially around dynamic dispatch, Java reflection, and any framework that resolves beans at runtime. Filter by confidence_score when it matters; do not treat inferred edges as ground truth in load-bearing tooling.
  • Hook loops. If the hook commits graphify-out/, and your CI then runs the hook again, you can trip rebuild storms. Easiest fix: [ -n "$GIT_AUTHOR_DATE" ] && exit 0 style guards, or just do not amend on the hook path.
  • Cache invalidation. The SHA256 cache keys on file content, not on extractor version. When you upgrade graphify, blow away graphify-out/cache/ before your first rebuild, otherwise you will mix old and new extractions silently.
  • Committing the viewer. graph.html is big and changes on every rebuild. In typical setups you .gitignore it and commit only graph.json and GRAPH_REPORT.md. Rebuild the viewer on demand.
  • Branch-scoped hooks and detached HEAD. git rev-parse --abbrev-ref HEAD returns HEAD when detached. The branch guard will correctly skip, which is what you want for a bisect session; just remember the graph will not track that.
  • Cost. Code extraction is local and cheap. The expensive pass is docs / PDFs / images, which calls the assistant'\''s model API. If your branch mostly changes code, prefer graphify update --no-viz (AST-only incremental) and reserve full re-extractions for doc changes.
  • AGENTS.md discipline. Commit it. The moment it lives only in one developer'\''s tree, the assistant behaviour diverges per machine and you lose the main reason to run this setup on a team.

Where this stops being enough

The branch-snapshot trick gets you AS OF <branch>. It does not get you AS OF <timestamp> or AS OF <commit> for arbitrary commits, because you only keep snapshots for branches you actually checked out. If you need a dense time axis (for example, to animate a full refactor across a hundred commits the way Figure 1 does with six), the honest answer is: script a git worktree, iterate commits, run graphify update --no-viz, archive the graph.json, move on. Minutes, not seconds, but perfectly tractable, and you only do it when you want the full film rather than the latest frame.

If query volume against graph.json becomes the bottleneck before repo scope does, graphify ships an MCP server (python -m graphify.serve graphify-out/graph.json, install extras via graphifyy[mcp]) that exposes query_graph, get_node, get_neighbors, and shortest_path as tools. The assistant stops shelling out to the CLI per question and starts calling a structured API against the snapshot you pointed it at — which composes nicely with the per-branch snapshot directory: one serve invocation per branch, wired into the assistant'\''s .mcp.json.

For anything heavier (cross-repo graphs, a queryable graph DB, continuous graph over a fleet of machines) you are out of graphify'\''s stated scope and into its sibling project, Penpax, or a real graph DB backend. --neo4j-push bolt://... is the obvious escape hatch when you outgrow graph.json but not the extraction pipeline. For a single-repo reporting PoC the lightweight per-branch snapshot pattern is the right amount of machinery.
